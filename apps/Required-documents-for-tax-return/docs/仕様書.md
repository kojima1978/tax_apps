# 確定申告 必要書類管理システム — 仕様書

## 1. 概要

### 1.1 目的
税理士事務所のスタッフが、確定申告に必要な書類リストを顧客・年度ごとに管理するためのWebアプリケーション。書類の準備状況をチェックリスト形式で管理し、印刷・Excel出力が可能。

### 1.2 ユーザー
税理士事務所のスタッフ（担当者）

### 1.3 動作環境
- モダンブラウザ（Chrome、Edge、Firefox、Safari）
- Docker環境（開発・本番）
- ローカル開発（Node.js 22以上）

## 2. 画面仕様

### 2.1 顧客ダッシュボード（/）

**概要**: トップ画面。登録済み顧客の一覧を表示し、各顧客の書類編集画面へ遷移する。

**表示要素**:
- ヘッダー: アプリ名「確定申告 必要書類案内」、ポータルリンク
- 検索バー: お客様名で部分一致検索
- 担当者フィルタ: プルダウンで担当者絞り込み
- 顧客カード一覧:
  - お客様名
  - 担当者名
  - 保存済み年度バッジ（令和N年）
  - 最終更新日
  - クリックで顧客詳細へ遷移
- お客様登録リンク
- 管理メニュー（折りたたみ式）:
  - 担当者一覧
  - 顧客一覧
  - データ管理
  - バックアップ・復元

### 2.2 顧客詳細（/customers/:id）

**概要**: 選択した顧客の年度別書類データを管理する。

**表示要素**:
- お客様名、担当者名
- 保存済み年度一覧（各年度をクリックで書類編集へ遷移）
- 新規年度作成フォーム（プルダウンで年度選択 → 作成・編集ボタン）
- お客様情報編集リンク

### 2.3 書類編集画面（/customers/:id/years/:year）

**概要**: メイン画面。カテゴリ別の書類チェックリストを表示・編集する。

#### ツールバー
| 要素 | 説明 |
|------|------|
| 戻るボタン | 顧客詳細画面へ戻る（未保存変更時は確認ダイアログ） |
| 保存ボタン | データをサーバーに保存（Ctrl+Sでも可能） |
| 変更を破棄 | サーバーの保存データで上書き |
| 標準に戻す | 初期書類リストにリセット |
| 情報バー | 対象年度、お客様名、担当者名 |
| 検索バー | 書類名でリアルタイムフィルタ（ハイライト表示） |
| 展開/折畳ボタン | 全カテゴリの展開・折りたたみ |
| 印刷レイアウト | 1列/2列切替 |
| 翌年度へコピー | 現在のデータを翌年度にコピー |
| 印刷 | ブラウザの印刷ダイアログ |
| Excel出力 | スタイル付きExcelファイルを生成・ダウンロード |
| JSON出力 | 顧客単位のJSON形式でエクスポート |
| JSON読込 | JSONファイルからインポート |

#### カテゴリ
- 展開/折りたたみ切替（Chevronアイコン）
- カテゴリ名の編集・削除
- ドラッグ&ドロップによる並び替え
- 全チェック / 全解除ボタン（カテゴリ内の全書類+小項目を一括操作）
- チェック済み件数 / 全件数表示（例: 3/5）

#### 書類アイテム
- チェックボックス（チェック済みは薄表示 + 取り消し線）
- テキスト編集（インライン）
- 小項目追加ボタン
- 編集・削除ボタン
- ドラッグ&ドロップによる並び替え

#### 小項目（サブアイテム）
- チェックボックス（チェック済みは薄表示 + 取り消し線）
- テキスト編集（インライン）
- 削除ボタン

#### 自動保存
- 変更後30秒経過で自動保存（サイレント）
- 保存成功時Toast通知「自動保存しました」
- 自動保存中に手動保存は行われない（排他制御）

#### 保存状態表示
- 「未保存」: 初期状態
- 「保存済み: HH:MM:SS」: 最後の保存時刻（緑色）
- 「保存エラー: メッセージ」: エラー時（赤色）

#### カテゴリ追加
- 新規カテゴリ名入力 → 追加
- 削除済み標準カテゴリの復元機能

### 2.4 担当者管理（/staff/）

**一覧**: 担当者名、携帯電話番号、紐付き顧客数

**作成・編集**:
- 担当者名（必須、ユニーク）
- 携帯電話番号（任意、ハイフンあり/なし対応）
- 「続けてお客様登録」ボタン（作成時）

**削除**: 紐付き顧客がいる場合は削除不可

### 2.5 顧客管理（/customers/）

**一覧**: お客様名、担当者名

**作成・編集**:
- お客様名（必須）
- 担当者（プルダウン選択）

### 2.6 保存データ管理（/data-management/）

**概要**: 全保存データの一覧表示・検索・削除。

**検索条件**: 担当者名、お客様名、年度
**ソート**: 担当者、お客様名、年度、更新日時
**ページネーション**: 20件/ページ
**インライン編集**: 顧客情報の変更
**削除**: 個別データの削除

## 3. データ仕様

### 3.1 書類データ構造（JSON）

```typescript
interface CategoryGroup {
  id: string;           // カテゴリID（例: "basic_info"）
  category: string;     // カテゴリ名（例: "基本情報・本人確認（必須）"）
  documents: DocumentItem[];
  note?: string;        // 備考（カテゴリ先頭に表示）
}

interface DocumentItem {
  id: string;           // 書類ID（例: "doc_1234567890"）
  text: string;         // 書類名
  checked: boolean;     // チェック状態
  subItems: SubItem[];  // 小項目リスト
}

interface SubItem {
  id: string;           // 小項目ID
  text: string;         // 小項目名
  checked: boolean;     // チェック状態
}
```

### 3.2 初期カテゴリ（標準リスト）

| # | カテゴリ名 | 書類数 |
|---|-----------|--------|
| 1 | 基本情報・本人確認（必須） | 3 |
| 2 | 【所得】事業（一般事業・フリーランスなど） | 12 |
| 3 | 【所得】医業・歯科医業 | 11 |
| 4 | 【所得】不動産収入（家賃・駐車場など） | 9 |
| 5 | 【所得】給与・年金・退職金 | 4 |
| 6 | 【所得】土地・建物や株式等の売却（譲渡） | 7 |
| 7 | 【所得】その他の所得 | 4 |
| 8 | 【控除】社会保険・生命保険・その他の控除 | 8 |
| 9 | 【控除】医療費控除 | 4 |
| 10 | 【控除】住宅ローン控除（初年度のみ） | 5 |
| 11 | 【控除】寄付金控除（ふるさと納税含む） | 3 |

### 3.3 年度表示
- 西暦→令和変換: `年 - 2018`（例: 2025年 → 令和7年）
- 書類テキスト内の `{year}` を `令和N年` に置換

## 4. 非機能要件

### 4.1 パフォーマンス
- 自動保存デバウンス: 30秒
- SQLite WALモード（読み書き並行処理）
- Viteによる高速ホットリロード
- N+1クエリ回避: 一覧取得系（顧客ダッシュボード、検索、バックアップ）は関連レコードを一括取得+`groupBy`でマッピング

### 4.2 データ保全
- beforeunload による未保存変更の警告
- 顧客削除時のCASCADE削除（関連する書類データも連動削除）
- 担当者削除時のSET NULL（紐付き顧客がいる場合はAPI側で拒否）
- バックアップ・復元機能（JSON全データ）
- 第3正規形 (3NF) に準拠したDB設計（担当者名はJOINで取得、冗長カラムなし）

### 4.3 印刷対応
- `@media print` によるレイアウト最適化
- 操作ボタン非表示（`.no-print`クラス）
- 1列 / 2列レイアウト切替
- 印刷専用ヘッダー（事務所情報・担当者携帯）
- 印刷専用フッター（注意事項・日付）

### 4.4 アクセシビリティ
- `role="toolbar"`, `role="radiogroup"` 等のARIAロール
- `aria-label`, `aria-checked`, `aria-live="polite"` の使用
- キーボード操作対応（Enter で確定、Escape でキャンセル）

## 6. デプロイ

### 6.1 Docker構成

| 対象 | Dockerfile | 備考 |
|------|-----------|------|
| フロントエンド | `../../docker/Dockerfile.vite-static`（共有） | 全Viteアプリ共通。`APP_PORT=3005`, `APP_PATH=tax-docs` |
| バックエンド | `backend/Dockerfile`（専用） | better-sqlite3ネイティブビルド対応 |

フロントエンド本番ビルド時に `VITE_API_URL=/tax-docs-api` を注入（docker-compose.prod.yml の `build.args` で設定）。

### 6.2 開発環境
```bash
docker compose up -d --build
```
- フロントエンド: http://localhost:3005/tax-docs/（Vite devサーバー、HMR対応）
- バックエンド: http://localhost:3006/api/（Express + ts-node-dev）

### 6.3 本番環境
```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
```
- フロントエンド: nginx配信（静的ファイル、非rootユーザー）
- バックエンド: tini + 非rootユーザー実行
- Nginx Gateway経由: http://localhost/tax-docs/

### 6.4 データ永続化
- SQLiteデータベース: Docker named volume `tax-docs-data`（`/app/data` にマウント）
- バックアップ: `/api/backup/export` でJSON全データ取得可能
